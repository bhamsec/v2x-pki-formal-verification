# Security and privacy requirements for the V2X PKI

The full P01–P106 list of security and privacy requirements derived via the TARA procedure (ISO/SAE 21434). The paper (Section 4) describes the derivation methodology; Section 6 references a subset of these when mapping ProVerif queries to requirements (see also [`RESULTS.md`](./RESULTS.md)). The rendered PDF version is in [`TARA/04_Security_privacy_requirements.pdf`](./TARA/04_Security_privacy_requirements.pdf).

Rows are grouped by the asset the requirement protects. Inline math renders on GitHub.

## ITS-S registration data

| ID | Requirement |
|----|-------------|
| P01 | ITS-S registration data must be encrypted and stored in a tamper-resistant HSM to guarantee confidentiality and integrity. Data shall remain within the HSM's secure boundary at all times. Confidentiality: $vid$, $Pri_V$, $profile_V$. Integrity: $vid$, $Pri_V$, $profile_V$, $(IP_{EA}, Cert_{EA})$, $(IP_{AA}, Cert_{AA})$, $Cert_{TrustedAAs}$. |
| P02 | Canonical ITS-S registration keys must be cryptographically strong: $Pri_V$, $Pub_V$. |
| P03 | The ITS-S must implement memory-protection mechanisms and access controls to prevent memory exhaustion or overload that could disrupt critical functionalities of the HSM. |
| P04 | EA registration data must be stored in protected memory to guarantee confidentiality and integrity. Data requiring confidentiality and integrity: $Pri_{EA}$. |
| P05 | EA keys must be cryptographically strong: $Pri_{EA}$, $Pub_{EA}$. |
| P06 | The EA must implement memory-protection mechanisms and access controls to prevent memory exhaustion or overload that could disrupt critical functionalities. |

## AA registration data

| ID | Requirement |
|----|-------------|
| P07 | AA registration data must be stored in protected memory to guarantee confidentiality and integrity. Data requiring confidentiality and integrity: $Pri_{AA}$. |
| P08 | AA keys must be cryptographically strong: $Pri_{AA}$, $Pub_{AA}$. |
| P09 | The AA must implement memory-protection mechanisms and access controls to prevent memory exhaustion or overload that could disrupt critical functionalities. |

## EA registration DB

| ID | Requirement |
|----|-------------|
| P10 | The registration database must enforce strict access controls and encryption to ensure that only the authorised EA can access or process sensitive information. |
| P11 | The registration database must implement physical security measures and access controls to prevent unauthorised tampering or manipulation of ITS-S registration entries. |
| P12 | The EA registration DB must implement memory-protection mechanisms and access controls to prevent memory exhaustion or overload that could disrupt critical functionalities. |

## Enrolment request

| ID | Requirement |
|----|-------------|
| P13 | Only registered and non-revoked ITS-Ss with valid canonical credentials must be allowed to perform the enrolment phase and obtain an enrolment certificate. |
| P14 | The EA must have a corresponding entry in the registration database for any ITS-S initiating the enrolment phase. |
| P15 | Information contained in the enrolment request must be encrypted. Data: $vid$, $Pub_{Enr}$, $attributes_{req}$, timestamp. Authorised readers: ITS-S and the EA that registered the ITS-S. |
| P16 | Only the authorised EA (i.e., the EA that registered the ITS-S) should have access to the real identity of the ITS-S performing the enrolment request, together with its $profile_V$ and the $attributes_{req}$ to be included in the enrolment certificate. |
| P17 | The ITS-S must exclusively perform the enrolment phase with the EA indicated during registration: $(IP_{EA}, Cert_{EA})$. |
| P18 | The ITS-S must not be linkable based on the EA it contacts for the enrolment phase. |
| P19 | The EA must be able to uniquely identify the ITS-S initiating the enrolment phase from the list of registered ITS-Ss. Data used to identify the ITS-S: $vid$. |
| P20 | Information contained in the enrolment request must be digitally signed by the ITS-S to ensure authentication and to prevent undetected modification. Signed data: $vid$, $Pub_{Enr}$, $attributes_{req}$, timestamp. |
| P21 | During the enrolment phase, the ITS-S must prove Proof of Possession of the canonical ITS-S registration private key $Pri_V$. |
| P22 | The enrolment request must contain a timestamp to prevent replay attacks. |
| P23 | The key used to decrypt the enrolment request must be accessible only to the legitimate EA, ensuring authentication of the recipient. |
| P24 | Enrolment keys generated by the ITS-S, whose public parts have to be included in the $Cert_{Enr}$, must be cryptographically strong: $Pub_{Enr}$, $Pri_{Enr}$. |
| P25 | Enrolment keys must be generated by the ITS-S with sufficient entropy to ensure a negligible probability of collision with other existing keys: $Pub_{Enr}$, $Pri_{Enr}$. |
| P26 | The ITS-S must prove Proof of Possession of the $Pri_{Enr}$ whose public part has to be included in the $Cert_{Enr}$. |
| P27 | The EA must employ redundancy, load balancing, and failover mechanisms to ensure continuous availability during high request loads or attempted disruptions for the enrolment phase. |
| P28 | Multiple enrolment requests and re-certification requests performed by the same ITS-S must not be linkable to each other. |

## Enrolment response

| ID | Requirement |
|----|-------------|
| P29 | Information contained in the enrolment response must be encrypted. Data: $Cert_{Enr} = \{pseudoid, Pub_{Enr}, attributes_{conf}, Sign_{EA}\}$. Authorised readers: ITS-S and the EA that registered the ITS-S. |
| P30 | Information contained in the enrolment response must be digitally signed by the designated EA to ensure authentication and to prevent undetected modification. Signed data: $pseudoid$, $Pub_{Enr}$, $attributes_{conf}$. |
| P31 | The enrolment response must be in a one-to-one relationship with the enrolment request to prevent replay attacks. |
| P32 | The ITS-S must stop waiting for the enrolment response after a predefined timeout following the request submission. |
| P33 | The key used to decrypt the enrolment response must be accessible only to the legitimate ITS-S that sent the request, ensuring authentication of the recipient. |
| P34 | The $pseudoid$ contained in the enrolment response must be generated by the EA with sufficient entropy to ensure a negligible probability of collision with other existing IDs. |

## Enrolment certificate

| ID | Requirement |
|----|-------------|
| P35 | The enrolment certificate issued to the ITS-S at the end of the enrolment phase must be digitally signed by the EA to ensure authentication and prevent undetected modification. |
| P36 | The enrolment certificate must contain attributes in line with the ITS-S $profile_V$. |
| P37 | The enrolment certificate must include a specific expiration time after which it must be renewed. |
| P38 | An ITS-S can only have one $Cert_{Enr}$ active at a time. |
| P39 | An ITS-S that already has a valid $Cert_{Enr}$ must not be issued with another one without first invalidating the previous one. |
| P40 | The EA must maintain a record of all issued enrolment certificates, including expired and renewed ones. |
| P41 | The EA must provide a way for the ITS-S to update the $Cert_{Enr}$ when it has expired or whenever the permissions of the ITS-S change. |
| P42 | A request for re-certification of the enrolment certificate must be indistinguishable from an initial enrolment request. |
| P43 | Before including the $Pub_{Enr}$ in the $Cert_{Enr}$ to be issued to the ITS-S, the EA must verify that this key has not already been certified for another ITS-S. |
| P44 | The $pseudoid$ to be included in the $Cert_{Enr}$ must not have already been certified for another ITS-S. |
| P45 | Only the designated EA must be allowed to link the issued enrolment certificate with the real identity of the ITS-S ($vid$). |

## Authorisation request

| ID | Requirement |
|----|-------------|
| P51 | Only enrolled and non-revoked ITS-Ss with valid and active enrolment credentials must be allowed to perform the authorisation phase and obtain ATs. |
| P52 | The EA must have an entry in the enrolment database for the enrolment certificate of any ITS-S initiating the authorisation phase. |
| P53 | Information contained in the authorisation request must be encrypted for both the designated AA and EA. Data for AA: $Pub_{AT}$, $services_{req}$, $Cert_{EA}$, timestamp. Data for EA: $services_{req}$, $\#(Cert_{Enr})$. |
| P54 | The ITS-S must exclusively perform the authorisation phase with the AA indicated during registration: $(IP_{AA}, Cert_{AA})$. |
| P55 | The ITS-S must not be linkable based on the AA it contacts for the authorisation phase. |
| P56 | Information contained in the authorisation request must be digitally signed by the ITS-S to ensure authentication and to prevent undetected modification. Signed data: $Pub_{AT}$, $services_{req}$, $Cert_{EA}$, timestamp, $\#(Cert_{Enr})$. |
| P57 | The authorisation request must contain a timestamp to prevent replay attacks. |
| P58 | The key used to decrypt the authorisation data intended for the AA must be accessible only to the legitimate AA, while the key used to decrypt the authorisation data intended for the EA must be accessible only to the legitimate EA, ensuring authentication of the recipient. |
| P59 | Authorisation data intended for the EA must be digitally signed using $Pri_{Enr}$ to provide Proof of Possession of that key. $Pub_{Enr}$ must be associated with a valid $Cert_{Enr}$ (e.g., not expired or revoked). Signed data for EA using $Pri_{Enr}$: $services_{req}$, $\#(Cert_{Enr})$. |
| P60 | Authorisation keys generated by the ITS-S, whose public parts have to be included in the AT, must be cryptographically strong: $Pub_{AT}$, $Pri_{AT}$. |
| P61 | Authorisation keys must be generated by the ITS-S with sufficient entropy to ensure a negligible probability of collision with other existing keys: $Pub_{AT}$, $Pri_{AT}$. |
| P62 | The ITS-S must prove Proof of Possession of the $Pri_{AT}$ whose public part has to be included in the AT. |
| P63 | The AA must employ redundancy, load balancing, and failover mechanisms to ensure continuous availability during high request loads or attempted disruptions for the authorisation phase. |
| P64 | The AA receiving the authorisation request must not be able to identify or profile the ITS-S based on the $services_{req}$ it is requesting. |
| P65 | Only the designated EA must be able to identify the ITS-S performing the authorisation phase, via the reference to its $Cert_{Enr}$ and the signature using $Pri_{Enr}$. Identifying the ITS-S means discovering its $vid$ and $profile_V$. |
| P66 | Multiple authorisation requests performed by the same ITS-S must not be linkable to each other. |

## Authorisation response

| ID | Requirement |
|----|-------------|
| P67 | Information contained in the authorisation response must be encrypted. Data: $AT = \{pseudoid, Pub_{AT}, services_{conf}, Sign_{AA}\}$. Authorised readers: ITS-S and the AA that issued the AT. |
| P68 | Information contained in the authorisation response must be digitally signed by the AA to ensure authentication and to prevent undetected modification. Signed data: $pseudoid$, $Pub_{AT}$, $services_{conf}$. |
| P69 | The authorisation response must be in a one-to-one relationship with the authorisation request to prevent replay attacks. |
| P70 | The ITS-S must stop waiting for the authorisation response after a predefined timeout following the request submission. |
| P71 | The key used to decrypt the authorisation response must be accessible only to the legitimate ITS-S that sent the request, ensuring authentication of the recipient. |
| P72 | The AA must not be able to issue a valid AT to the ITS-S without first interacting with the EA. |
| P73 | $services_{conf}$ inserted in the AT issued by the AA to the ITS-S must be those authorised by the designated EA, and in line with the attributes contained in the ITS-S enrolment certificate. |
| P74 | The ITS-S should be able to verify that $services_{conf}$ inserted in the AT comes from the designated EA. |
| P75 | The $pseudoid$ contained in the authorisation response must be generated by the AA with sufficient entropy to ensure a negligible probability of collision with other existing IDs. |

## Authorisation ticket

| ID | Requirement |
|----|-------------|
| P76 | The AT issued to the ITS-S at the end of the authorisation phase must be digitally signed by the AA to ensure authentication and prevent undetected modification. |
| P77 | The AT must include a short specific expiration time after which it must be changed. |
| P78 | Multiple ATs issued by the AA to the same ITS-S for the same V2X service must have consecutive validity periods. |
| P79 | Before including the $Pub_{AT}$ in the AT to be issued to the ITS-S, the AA should verify that this key has not already been certified for another ITS-S (by the AA itself or by any other AAs). |
| P80 | The $pseudoid$ to be included in the AT must not have already been certified for another ITS-S. |
| P81 | The $pseudoid$ to be included in the AT must not be linkable to the real identity of the ITS-S ($vid$). Exception: authorised AAs and EAs. |
| P82 | The authorised EA and AA, by cooperating, must be able to disclose the identity of the ITS-S associated with a specific AT. |
| P83 | Only the designated AA must have knowledge of the AT issued to the ITS-S as a result of the authorisation phase, before the ITS-S decides to use it. |

## Authorisation validation request

| ID | Requirement |
|----|-------------|
| P86 | Information contained in the authorisation validation request must be encrypted for the designated EA. Data for EA: $services_{req}$, $\#(Cert_{Enr})$, timestamp. Authorised readers: AA and EA. |
| P87 | Information contained in the authorisation validation request must be digitally signed by the AA to ensure authentication and to prevent undetected modification. Signed data for the EA: $services_{req}$, $\#(Cert_{Enr})$, timestamp. |
| P88 | The authorisation validation request must contain a timestamp to prevent replay attacks. |
| P89 | The key used to decrypt the authorisation validation request must be accessible only to the legitimate EA, ensuring authentication of the recipient. |
| P90 | The AA must forward the authorisation validation request to the correct EA designated by the ITS-S in the authorisation request. This must be the same EA that enrolled the ITS-S. |
| P91 | The EA must employ redundancy, load balancing, and failover mechanisms to ensure continuous availability during high request loads or attempted disruptions for the authorisation phase. |
| P92 | Multiple authorisation validation requests addressing the same ITS-S must not be linkable to each other. |

| ID | Requirement |
|----|-------------|
| P93 | Information contained in the authorisation validation response must be encrypted. Data: $services_{conf}$. Authorised readers: AA and the EA designated by the ITS-S. |
| P94 | Information contained in the authorisation validation response must be digitally signed by the EA to ensure authentication and to prevent undetected modification. Signed data: $services_{conf}$. |

## Unicast V2X messages

| ID | Requirement |
|----|-------------|
| P99 | The content of a confidential V2X unicast message must be encrypted to ensure that only the intended recipient can decrypt it. |

## Broadcast V2X messages

| ID | Requirement |
|----|-------------|
| P100 | It must not be possible for any node in the system to deduce the location and the persistent ITS identity of an end-user by analysing broadcast V2X messages. Persistent ITS identities: $vid$ and $pseudoid$ contained in the $Cert_{Enr}$. Exception: AA and EA, but they must collaborate. |
| P101 | An ITS-S must not include the persistent ITS end-user identity within broadcast V2X messages. These messages must instead include a pseudonymous identity temporarily associated exclusively with the ITS-S. Pseudonymous temporary identity: $pseudoid$ contained in ATs. |

## Unicast and broadcast V2X messages

| ID | Requirement |
|----|-------------|
| P102 | Information contained in broadcast or unicast V2X messages must be digitally signed by the sender ITS-S to ensure authentication and to prevent undetected modification. Signed using $Pub_{AT}$ of the currently active AT. |
| P103 | An ITS-S can only have one AT per specific service active at a time. |
| P104 | Active ATs attached to V2X messages must be changed regularly to prevent long-term linkability. |
| P105 | V2X messages must contain a timestamp to prevent replay attacks. |
| P106 | Multiple V2X messages sent by the same ITS-S using different ATs must not be linkable to each other. |
